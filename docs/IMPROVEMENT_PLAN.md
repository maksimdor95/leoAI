## 1. Контекст: что ломало UX

| Симптом | Вероятная причина (по коду) |
|--------|------------------------------|
| «Вакансии (0)» при том, что раньше были | Порог матча `score >= 30`; неполный `collectedData` в Redis; авто-запрос матча после каждого сообщения при «скудном» профиле даёт пустой топ |
| Зацикливание на «сколько мест описать» при ответах `3` / `три` | Ответ проходит через LLM `validateAnswer`; короткие числовые ответы помечаются как `unclear`, шаг не сохраняется |
| CORS / `net::ERR_FAILED` на логин при `127.0.0.1` vs `localhost` | Разные origin; до исправления в `user-profile` был один `CORS_ORIGIN` |
| `401 Invalid token` на `/api/jobs/match` | Расхождение `JWT_SECRET` или чтение env до `dotenv` (частично исправлено в `authService` / `jwt`) |
| Конфликт портов (`EADDRINUSE`), сервисы на `8080` | Дефолт `PORT=8080` в нескольких сервисах при запуске без `PORT=` |
| Кнопка «Обновить» в панели вакансий | Осознанный fallback; для «чистого» UX можно убрать или показывать только при ошибке |

## 2. Приоритет P0 — устранение «тупых» блокировок и пустых вакансий

Цель: пользователь не залипает на анкете и понимает, почему список вакансий пуст.

### 2.1 Детерминированный приём ответов для «числовых» шагов сценария

- **Статус (2026-04):** сделано для `positionsCount` — локальный парсер в `services/conversation` (`numericStepAnswers.ts` + `dialogueEngine`), успешный парс без `validateAnswer`, в `collectedData` пишется число.
- **Проблема:** `positionsCount` и аналогичные шаги не должны зависеть от мнения LLM о «достаточной подробности» числа.
- **Действия:**
  1. В `conversation` (например, `dialogueEngine` или слой перед `validateAnswer`): для шагов с известным `collectKey` из whitelist (`positionsCount`, возможно `totalExperience` в формате «N лет») — **парсить локально** (цифры + слова «один»…«пять», диапазон 1–5).
  2. При успешном парсе — **сразу писать в `collectedData`** и переходить к следующему шагу **без** вызова `validateAnswer`, либо вызывать валидатор только если парсер не уверен.
  3. В `ai-nlp`: дополнить промпт `validate-answer` правилом: «если вопрос про количество позиций 1–5, ответ «3»/«три» достаточен» — как второй слой защиты.
- **Критерий готовности:** ответы `3`, `три`, `2-4`, `два` проходят с первого раза; нет повторов одного и того же вопроса без смены формулировки сценария.

### 2.2 Пустой подбор: прозрачность для пользователя и мягче порог (поэтапно)

- **Статус (2026-04):** фаза A частично внедрена — API `GET /api/jobs/match` отдаёт `jobsInDb`, `jobsScanned`, `maxMatchScore`, `matchThreshold`; в логах job-matching строка `[match]` и предупреждение, если в БД есть вакансии, но ни одна не выше порога; на фронте (чат Jack) пустой список объясняется текстом с порогом и «лучшим баллом», под заголовком показан объём каталога.
- **Проблема:** жёсткий отсрез `score >= 30` без объяснения даёт ощущение «вакансий нет на рынке».
- **Действия (фаза A — быстро):**
  1. В ответе API или на фронте: если `totalMatched === 0` при ненулевом числе вакансий в БД — показывать **«Подходящих по профилю пока нет; покажем ближайшие, если заполните …»** (короткий текст, без технических деталей).
  2. Логировать на бэкенде (job-matching): `count in DB`, `count after threshold`, топ-1 score если `< 30` (только в логи).
- **Действия (фаза B — позже):**
  - Ввести **динамический порог** или блок «Слабое совпадение» с низким score отдельно от «Рекомендуем».
- **Статус фазы B (2026-04):** второй ярус в `job-matching`: диапазон `[WEAK_MATCH_SCORE_FLOOR, MATCH_SCORE_THRESHOLD)` — до 15 позиций в ответе (`weakJobs`, `weakTierTotal`, `weakMatchFloor`); на фронте Jack — секции **«Рекомендуем»** и **«Слабое совпадение»** с пояснением порогов. Динамический порог по пользователю пока не вводился — при необходимости отдельно.
- **Критерий готовности:** пользователь видит отличие «рынок пуст» vs «профиль ещё не заполнен / порог высокий».

### 2.3 Частота авто-матча и бесполезные тосты

- **Статус (2026-04):** на фронте Jack — авто-запрос `/api/jobs/match` после сообщения и при открытии вкладки «Вакансии» только если в `collectedData` выполнен минимум (`frontend/lib/jackProfileGating.ts`: `desired_role` / `desiredRole`, либо навыки + `totalExperience`, либо любое `position_N_role` + `totalExperience`). **`metadata` сессии обновляется в `chatApi` после каждого `sendMessage`, `executeCommand` и при поллинге**, чтобы критерий был актуален. Ручная кнопка пересчёта по-прежнему всегда дергает API. Явный refresh: тост при нуле вакансий различает пустой каталог и «есть в БД, но ниже порога»; при `silent` тостов нет.
- **Проблема:** автозапрос после каждого сообщения при ранней анкете даёт много «0 вакансий» и info-тостов.
- **Действия:**
  1. Не вызывать авто-матч, пока не собран минимальный набор ключей (например есть `desiredRole` или `skills` + `totalExperience`), **или**
  2. При `jobs.length === 0` в silent-режиме **не показывать тост** (только обновить панель), полноценное уведомление — только при `jobs.length > 0` или по кнопке «Обновить».
- **Критерий готовности:** ранний диалог не заспамлен «нет вакансий».

## 3. Приоритет P1 — надёжность окружения и DX

### 3.1 Единые порты по умолчанию

- **Действия:** в локальных `.env` (по сервисам) и в коде заменить дефолт `8080` на целевые: `3001` user-profile, `3002` conversation, `3003` ai-nlp, `3004` job-matching, `3005` email; либо документировать обязательный `PORT` при `npm run dev`.

### 3.3 CORS для локальной разработки

- **Статус:** разрешены `http://localhost:3000` и `http://127.0.0.1:3000`.
- **Дополнительно:** при появлении preview-URL (ngrok и т.д.) — вынести список в `CORS_ORIGIN` через запятую или regex в dev-only.

## 8. Как читать приоритеты и «связки» вне одного пункта

- **Нумерация P0 / P1 / P2** — это очерёдность *риска для UX* и *зависимостей*, а не один файл в коде на пункт. Несколько файлов могут закрывать один пункт; один коммит может закрывать куски разных пунктов.
- **Раздел 6** задаёт календарный порядок (неделя 1 → неделя 2). Его можно отклонять, если продуктовая боль очевидна (например, сначала прозрачность вакансий, потом тосты).
- **Связка «P0.2 + кнопка обновления + разнообразие матча»** из обсуждения мапится так:
  - **Прозрачность пустого / слабого подбора** → **§2.2** (фаза A: тексты, метрики в ответе API, логи).
  - **Что делает кнопка у списка** → **§4.3** (UX: не путать пересчёт матча с загрузкой новых вакансий на сервер).
  - **Нормализация полей анкеты и tie-break** → **§4.2** (один слой перед матчем, честнее score и порядок при равенстве).
